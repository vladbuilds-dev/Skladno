# Складно: проверка слабых мест (07.07.2026)

## 1. «Данные важнее математики» — риск подтверждён, но управляем

Симуляция Монте-Карло (200 SKU, оборот 5 млн ₽/мес, 8 складов-кластеров, пуассоновский спрос, 60 дней истории, план на 30 дней):

- Медианная ставка спроса SKU×регион — **0,026 ед/день**; **83% ячеек** имеют ставку <0,1 ед/день. Прямой прогноз по ячейке даёт медианную ошибку **~100%** — предчувствие верное: наивный прогноз SKU×регион — мусор.
- Но экономика ошибок ограничена. Стоимость ошибок плана за 30 дней (медиана, ₽):

| Стратегия | Ошибки плана | к оракулу |
|---|---|---|
| Оракул (истинный спрос) | 59 570 | — |
| Наивный прогноз SKU×регион | 90 416 | +52% |
| Пул: ставка SKU × доли регионов портфеля | 78 523 | +32% |
| Байесовский бленд (ячейка ↔ пул) | 72 270 | +21% |
| Всё на один склад (Коледино) | 189 945 | +219% |

**Выводы:**
1. Даже с шумным прогнозом оптимизированное распределение в 2–2,5 раза дешевле статус-кво «всё на один склад». Ценность продукта переживает шум данных.
2. Иерархический прогноз (ставка SKU × доли регионов, с блендингом) — обязателен. Наивный per-cell прогноз есть смысл сразу исключить из архитектуры.
3. Против реального конкурента (Поставыч-стиль: гео-распределение) чисто прогнозного преимущества почти нет (+32% vs +21%). **Преимущество Складно — не «лучше прогнозируем», а слой издержек: коэффициенты, хранение, нелокальность, бюджет, фура.** Это меняет позиционирование.

## 2. Волатильность коэффициентов — подтверждена, жёстче ожиданий

- Коэффициенты обновляются **каждые несколько часов**; в высокий сезон приёмка на большинстве складов недоступна или по коэффициентам **до ×20** (в заметке — «×6 и выше»).
- Слоты появляются волнами (сброс броней ~2:00–4:00 и ~7:00 МСК). Официальный API `/acceptance/coefficients` отдаёт календарь на 14 дней, лимит 6 запросов/мин.
- Следствие: статический план устаревает за часы. Продукт должен выдавать **политику, а не точку**: топ-3 альтернативных плана + порог коэффициента («грузите на Тулу при К≤2, иначе Казань»), пересчёт при изменении календаря. Это усложняет MVP относительно «выгрузка → план → Excel».

## 3. Данные из API — есть всё необходимое (риск снят)

- `/orders`: каждый заказ с `regionName`, `oblastOkrugName`, `warehouseName` — сырьё для гео-спроса есть.
- `/acceptance/coefficients`: календарь на 14 дней по складам и типам коробов.
- API тарифов: хранение, логистика. Остатки по складам — отчёт warehouseRemains.

## 4. Окно рынка — риск консолидации реален

- Точка купила 100% MPStats за ~2 млрд ₽ (февраль 2026), выручка MPStats 1,8 млрд ₽ (2025), 40 000+ селлеров WB. Это уже **третья** e-com сделка Точки (EcomCom 2023, 24AI 2024). Банки целенаправленно собирают экосистему для селлеров.
- Родные инструменты кабинета WB + MP Manager и др. уже делают распределение по складам с учётом оборачиваемости и перераспределение остатков. Арифметически, без совместной оптимизации — пробел пока держится, но окно 12–24 месяца, не больше.

## Итоговый вердикт

Идея жива, но с двумя поправками к плану:
1. **Продавать не прогноз, а стоимость логистики.** Прогнозный слой — иерархический и скромный; вся дифференциация — в MILP по издержкам.
2. **MVP — не файл, а политика с пересчётом.** Волатильность коэффициентов делает разовый Excel-план недостаточным; закладывать пересчёт/алерты с первого дня (это же и удержание для подписки).
3. Скорость критична: консолидация подтверждена сделкой на 2 млрд ₽ — 90-дневный план валидации не «желателен», а обязателен.

Симуляция: `sim_demand.py` (в этой же папке можно воспроизвести: python3 sim_demand.py).
